Three kinds of question, answered with receipts

Every answer resolves to stored evidence rows — or to a machine-readable refusal that says exactly what is not measured, why, and what proxies exist. Never silence, never interpolation.

Causal

"Why did Norris lose the British GP?"

One fetch returns the event dossier: result and grid rows, pit stops, retirements, weather at the circuit, standings before the round, market prices — each row carrying its source.

Factor families that aren't covered for the event are listed as explicit gaps in the same payload.
Comparative

"Hamilton vs Schumacher — career rates?"

Athlete cards carry both provenance layers, labeled: source-published career totals AND rollups re-derived from stored race rows, cross-checked on every build.

When the two layers disagree, the discrepancy is logged and citable — never silently corrected.
Attribute

"How fast does Alex Eala serve?"

No licensed keyless source measures serve speed — for any player. So the answer is a first-class refusal object naming the gap, plus honest proxies (ace rate, first-serve percentages) derived from cited match rows.

Refusals are data, not errors. Coverage is declared per sport, per factor family, in the public packs.

Facts in, interpretation never

The whole store reduces to one row shape: one fact, from one source, about one subject, in one context. Everything else — form, head-to-heads, era comparisons — is derived at question time with citations back to rows like this one.

The evidence row

Tidy NDJSON, one fact per line. License grade lives in the public register, never on rows; sources graded non-commercial or blocked never enter any tier — enforced mechanically at build time.

{"date":"2026-07-05","sport":"f1","span":"event",
 "event":"f1/2026/wc/r12-british-gp/race",
 "entity":"athlete/lando-norris",
 "metric":"result.position","value":2,"unit":"rank",
 "source":"f1db"}

For agents (and their humans)

Three front doors, same data: a plain URL for anything that speaks HTTP, an llms.txt for anything that reads, and — from Phase 4 — a typed MCP server for anything that calls tools.

Plain HTTP

Every payload is a static file. Event dossiers, athlete cards, schedules, packs — one envelope everywhere.

curl -s https://cdn.jsdelivr.net/gh/forge-arcana/kos-data@main/catalog.json \
  | jq .datasets
  • catalog.json — machine-readable index
  • status.json — calendar-aware freshness
  • license-register.json — every source, graded, with obligations

llms.txt

One URL that tells a language model everything — including, per sport, the coverage gaps: what KOS cannot show, declared before you ask.

https://cdn.jsdelivr.net/gh/forge-arcana/kos-data@main/llms.txt
  • Read it in the browser →

MCP server

kos-mcp ships with Phase 4: typed tools for events, athletes, comparisons and attributes — refusal objects included, reading this CDN.

{ "mcpServers": { "kos": {
    "command": "npx", "args": ["kos-mcp"]
} } }
  • find_entity · get_event · explain_result · compare_entities · get_attribute · get_coverage

Public · free

Hot tier — current season + 90 days

Event dossiers for the current season plus a rolling 90-day window, and athlete/team cards for every entity with data — always public, always current, served free on this CDN. Registries, packs, schedules and the license register ride along.

Private · offered separately

Deep history — the full archive

Every season back to the start of each sport, plus the forward-accumulated state series (rankings, ratings, availability, market prices over time) that can never be backfilled. One tidy row each, with annual NDJSON.gz and Parquet roll-ups. The format is public: observations.schema.json. Interested? Open an issue.
